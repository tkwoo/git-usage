Agent 기반으로 작업하기 위한 Git, GWS 사용 방법

브랜치, PR 초안, 팀 Git Skill, gws Calendar/Drive 자동화 명령, 통합 agent 프롬프트를 직접 만든다.

강의 성격

듣는 강의가 아니라 따라 하는 실습

목표는 Git과 gws 명령어를 외우는 것이 아니라, agent에게 맡길 수 있는 작업 단위를 안전하게 설계하는 것이다.

  • 설명은 각 실습 시작 전 1~2분만 한다.
  • 각 파트마다 산출물을 하나씩 남긴다.
  • gws 인증이 안 된 사람은 실제 실행 대신 명령과 payload를 작성한다.
  • push, PR 생성, Calendar 생성, Drive 권한 변경은 사람 확인 뒤 진행한다.

Part 0

실습 환경 확인

agent에게 일을 시키기 전에 현재 작업장이 안전한지 확인한다. 이미 변경된 파일이 있으면 agent가 실수로 섞어 바꿀 수 있다.

  • 실습용 issue 번호를 하나 정한다. 예: #23
  • 현재 작업 디렉터리가 맞는지 확인한다.
  • 작업 전 변경 파일이 있으면 건드려도 되는지 판단한다.
  • 오늘 실습에서는 main에 직접 push하지 않는다.
실습 0-A

내가 직접 확인할 명령

이 네 가지는 agent 작업 전 기본 점검이다. 특히 git status --short는 의도하지 않은 파일이 섞였는지 가장 빠르게 보여준다.

pwd
git status --short
git branch --show-current
git remote -v

성공 기준: 현재 branch와 변경 파일 목록을 말할 수 있어야 한다.

실습 0-B

Agent에게 상태 확인 맡기기

agent에게도 바로 수정하라고 하지 말고, 먼저 상태를 읽고 위험 요소를 요약하게 한다.

현재 저장소의 git 상태를 확인해줘.
다음을 요약해줘.

1. 현재 branch
2. 변경된 파일 목록
3. main에서 직접 작업 중인지 여부
4. 실습 전에 주의해야 할 점
Part 1

Git flow는 이 한 줄로 압축한다

팀 협업에서 중요한 것은 복잡한 브랜치 전략 이름이 아니라, 변경이 추적되고 리뷰 가능한 상태로 이동하는 것이다.

issue -> branch -> change -> diff -> commit -> PR -> review -> merge
  • agent에게 일을 맡겨도 흐름은 바뀌지 않는다.
  • agent가 바꾼 내용도 PR 전 사람이 읽는다.
  • branch 이름과 PR 설명은 팀 협업의 검색 키가 된다.
Branch 규칙

issue 번호를 branch 이름에 넣는다

branch 이름은 작업의 출처를 알려주는 최소한의 메타데이터다. issue 번호가 있으면 PR, 회의, 일정, Drive 자료를 같은 맥락으로 묶을 수 있다.

작업branch예시
기능feat/{issue-number}feat/8
버그fix/{issue-number}fix/15
문서docs/{issue-number}docs/23
리팩터링refactor/{issue-number}refactor/31
실습 1-A

Agent에게 branch 준비시키기

좋은 요청은 목표, branch 이름, 금지 사항, 사전 확인을 함께 포함한다. “브랜치 만들어줘”보다 아래처럼 제약을 명시해야 안전하다.

wel-bridge 저장소에서 issue #23 문서 작업 branch를 준비해줘.
branch는 docs/23으로 만들고 main에는 직접 push하지 마.

작업 전:
1. git status를 확인해.
2. 현재 branch를 확인해.
3. 변경 파일이 있으면 먼저 알려줘.

branch 생성 후 현재 branch를 보여줘.
실습 1-B

직접 실행할 경우

agent가 실행하든 사람이 실행하든 기준은 같다. 최신 main에서 새 작업 branch를 만들고, 현재 branch를 확인한다.

git switch main
git pull origin main
git switch -c docs/23
git branch --show-current

성공 기준: 현재 branch가 docs/23으로 출력된다.

Git worktree

한 저장소에서 여러 작업 디렉터리를 쓴다

git worktree는 같은 Git history를 공유하면서 branch별 working tree를 따로 만든다. 현재 작업을 건드리지 않고 agent에게 별도 작업을 맡길 때 유용하다.

  • 현재 작업 중인 변경을 보존한 채 다른 branch를 열 수 있다.
  • 문서 작업, 기능 작업, 리뷰 확인을 동시에 진행할 수 있다.
  • agent에게 작업 경로를 명확히 지정할 수 있다.
실습 1-C

worktree 기본 사용법

새 디렉터리에 docs/23 branch를 만들고, 그 경로에서만 agent 작업을 진행한다.

git worktree list
git worktree add ../wel-bridge-docs-23 -b docs/23 main
cd ../wel-bridge-docs-23
git branch --show-current
git status --short

정리할 때는 git worktree remove ../wel-bridge-docs-23를 사용한다.

worktree 주의점

agent에게 경로를 정확히 알려야 한다

  • 같은 branch를 두 worktree에서 동시에 checkout할 수 없다.
  • worktree 디렉터리를 수동 삭제하기보다 git worktree remove를 사용한다.
  • agent 요청에는 작업할 worktree 경로를 명시한다.
현재 작업 디렉터리는 건드리지 말고,
../wel-bridge-docs-23 경로의 worktree에서 issue #23 문서 작업을 진행해줘.
Git sparse checkout

필요한 경로만 working tree에 펼친다

git sparse-checkout은 저장소 전체가 아니라 선택한 폴더만 보이게 한다. 큰 저장소나 monorepo에서 특정 서비스 또는 문서 폴더만 작업할 때 유용하다.

  • agent가 건드릴 수 있는 작업 표면을 줄인다.
  • clone과 checkout 시간을 줄일 수 있다.
  • 문서 실습처럼 일부 경로만 필요한 상황에 적합하다.
실습 1-D

sparse checkout 기본 사용법

이미 clone한 저장소에서 필요한 폴더만 펼친다. 실습에서는 예시로 ideas, meetings만 사용한다.

git sparse-checkout init --cone
git sparse-checkout set ideas meetings
git sparse-checkout list
git sparse-checkout disable
sparse checkout 주의점

보안 기능이 아니라 작업 범위 제어다

  • sparse checkout은 숨긴 파일을 권한상 보호하는 기능이 아니다.
  • 빌드나 테스트가 펼쳐지지 않은 경로에 의존하면 실패할 수 있다.
  • agent에게 현재 sparse checkout 범위를 알려줘야 파일 누락을 오해하지 않는다.
이 저장소는 sparse checkout으로 ideas와 meetings 폴더만 펼쳐서 작업해줘.
다른 경로가 필요하면 바로 수정하지 말고, 필요한 경로를 먼저 알려줘.
Part 2

PR은 변경을 설명하는 문서다

PR은 단순히 merge 버튼을 누르기 위한 절차가 아니다. 팀원이 변경의 의도와 위험을 검토할 수 있게 만드는 협업 문서다.

  • 무엇을 바꿨는지
  • 왜 바꿨는지
  • 무엇을 검증했는지
  • 리뷰어가 어디를 봐야 하는지
Commit 규칙

Conventional Commit으로 의도를 남긴다

commit prefix는 변경의 종류를 빠르게 읽게 해준다. agent가 commit 메시지를 만들 때도 팀 prefix를 지키게 해야 한다.

docs: add PR collaboration guide
feat: add calendar automation script
fix: handle missing drive folder id
chore: update gws setup notes
  • 하나의 commit은 하나의 의도를 가진다.
  • 생성 파일, 포맷팅, 기능 변경은 가능하면 분리한다.
실습 2-A

diff를 사람이 읽는다

git diff는 agent 작업의 검수 지점이다. 예상한 파일만 바뀌었는지, 삭제나 포맷팅 변경이 섞였는지 확인한다.

git status --short
git diff
git log --oneline --decorate -5

agent가 변경했더라도 최종 책임은 사람에게 있다.

실습 2-B

PR 설명 초안 만들기

agent에게 PR 본문 초안을 맡기면 빠르다. 단, 초안은 그대로 붙이지 말고 diff와 요구사항에 맞는지 사람이 고친다.

현재 변경 사항을 기준으로 PR 설명 초안을 작성해줘.
다음 섹션을 포함해줘.

좋은 PR 템플릿은 리뷰어의 시간을 아낀다. 특히 검증과 리뷰 포인트는 agent가 놓치기 쉬운 맥락을 사람이 보강해야 한다.

Part 3

팀 규칙을 Skill로 만든다

매번 긴 프롬프트로 규칙을 반복하지 않고, agent가 branch/commit/PR 작업에서 같은 규칙을 참고하게 만든다.

Skill은 agent를 팀에 온보딩하는 짧은 작업 매뉴얼이다.

Skill에 넣을 내용

일반 Git 설명보다 팀 고유 규칙

모델은 Git 자체는 이미 안다. Skill에는 “우리 팀에서는 어떻게 해야 하는지”와 “절대 하면 안 되는 일”을 넣는다.

  • 스킬 이름과 trigger 설명
  • branch naming 규칙
  • commit message prefix
  • PR 생성 전 체크리스트
  • 금지 작업
  • 검증 방법
실습 3-A

Skill 생성 프롬프트

규칙을 자연어로 먼저 정리한 뒤, agent에게 SKILL.md 형식으로 바꾸게 한다.

우리 팀의 Git 협업 규칙을 Codex Skill로 만들어줘.
스킬 이름은 team-git-workflow로 하고, 다음 규칙을 포함해줘.

- main 직접 작업 금지
- issue 번호 기반 branch naming: feat/8, fix/15, docs/23
- commit prefix: feat, fix, docs, refactor, test, chore
- PR 생성 전 git status, git diff, 테스트 확인
- PR 템플릿 포함
- credential, .env, token commit 금지

SKILL.md는 간결하게 작성해줘.
실습 3-B

Skill 파일 구조

description은 가장 중요하다. agent가 언제 이 Skill을 불러야 할지 판단하는 trigger 문장이다.

team-git-workflow/
└── SKILL.md
---
name: team-git-workflow
description: Use when working in the team's Git repository, creating branches,
making commits, opening PRs, or asking an agent to follow team PR and branch rules.
---
실습 3-C

Skill 본문 예시

본문은 길 필요가 없다. 작업 전 반드시 확인할 규칙과 체크리스트만 남긴다.

# Team Git Workflow

Use this skill before changing code or documentation in the team repository.

## Branch Rules

- Do not work directly on main.
- Start from the latest main.
- Branch names must use an issue number.

## PR Rules

- Run git status --short.
- Review git diff.
- Fill out the PR template.
Skill 검증

이 질문에 답할 수 있어야 한다

Skill 작성은 파일 생성으로 끝나지 않는다. 실제 위험 요청을 받았을 때 올바르게 멈추거나 확인 질문을 하는지 점검한다.

  • main에 바로 작업해줘를 막을 수 있는가?
  • issue 번호가 없는 branch 요청에서 확인 질문을 하는가?
  • PR 생성 전에 diff와 테스트 확인을 요구하는가?
  • 팀원이 읽어도 바로 적용할 수 있을 만큼 간결한가?
Part 4

gws CLI로 Workspace를 자동화한다

Calendar와 Drive 작업을 UI 클릭이 아니라 명령어와 JSON payload로 관리한다.

agent에게 Workspace 작업을 맡길 때는 실행보다 payload 검토가 먼저다.

gws 준비

설치와 인증

gws는 Google Workspace API를 CLI로 호출한다. 로컬 실습에서는 OAuth 인증을 사용하고, 필요한 scope만 승인한다.

npm install -g @googleworkspace/cli
gws auth setup
gws auth login --scopes drive,calendar
  • 인증이 안 되면 실제 실행 대신 payload 작성까지만 진행한다.
  • credentials.json, .env, token은 commit하지 않는다.
gws 안전 규칙

Workspace 자동화는 권한 확인이 핵심

Git 실수는 revert할 수 있지만, 잘못된 일정 초대나 Drive 공유는 팀 외부로 바로 노출될 수 있다.

  • 처음에는 필요한 scope만 승인한다. 예: drive,calendar
  • Calendar 생성 전 참석자, 시간대, 설명을 확인한다.
  • Drive 공유 전 파일 ID, 대상 이메일, role을 확인한다.
  • agent에게 바로 실행시키지 말고 payload를 먼저 보여 달라고 한다.
실습 4-A

Calendar 목록 확인

일정을 만들기 전에 어떤 캘린더에 만들지 확인한다. 개인 캘린더와 팀 캘린더를 혼동하지 않도록 한다.

gws calendar calendarList list
gws calendar +agenda

산출물: 어떤 캘린더에 만들지 결정한다.

실습 4-B

PR 리뷰 일정 만들기

helper command는 실습에서 이해하기 쉽다. 시간은 ISO 8601 형식으로 쓰고, 한국 시간은 +09:00을 명시한다.

gws calendar +insert \
  --summary "wel-bridge PR review" \
  --start "2026-05-07T19:00:00+09:00" \
  --end "2026-05-07T19:30:00+09:00" \
  --attendee teammate@example.com \
  --description "PR 리뷰 및 merge 기준 확인"
실습 4-C

Calendar API payload

agent에게 실제 실행을 맡기기 전에는 이런 JSON payload를 먼저 보여 달라고 한다. 참석자와 시간대가 맞는지 사람이 확인한다.

gws calendar events insert \
  --params '{"calendarId": "primary"}' \
  --json '{
    "summary": "wel-bridge PR review",
    "start": {"dateTime": "2026-05-07T19:00:00+09:00", "timeZone": "Asia/Seoul"},
    "end": {"dateTime": "2026-05-07T19:30:00+09:00", "timeZone": "Asia/Seoul"},
    "attendees": [{"email": "teammate@example.com"}]
  }'
Part 5

Drive에 코드 관련 산출물을 올린다

PR 리뷰 자료, 테스트 로그, 회의록을 Drive에 모으면 리뷰 일정과 PR 본문에서 같은 자료를 참조할 수 있다.

  • 테스트 결과 로그
  • 회의록 Markdown
  • PR 변경 요약
  • API 응답 샘플 JSON
  • 분석용 CSV
실습 5-A

Drive 파일과 폴더 확인

먼저 최근 파일을 확인하고, 실습용 폴더가 없다면 만든다. 폴더 ID는 이후 업로드 명령의 --parent에 사용한다.

gws drive files list \
  --params '{"pageSize": 10, "orderBy": "modifiedTime desc"}'
gws drive files create --json '{
  "name": "wel-bridge-review-assets",
  "mimeType": "application/vnd.google-apps.folder"
}'
실습 5-B

파일 업로드 명령

파일명에는 날짜, PR 또는 issue 번호를 넣는다. 나중에 Drive에서 검색할 때 맥락을 바로 알 수 있어야 한다.

gws drive +upload ./test-results/report.json \
  --parent FOLDER_ID \
  --name "20260507-pr23-test-report.json"

산출물: 업로드할 파일명과 대상 폴더 ID.

실습 5-C

공유 권한 payload

공유 권한은 가장 조심해야 한다. 실습에서는 읽기 권한부터 사용하고, 대상 이메일과 파일 ID를 반드시 확인한다.

gws drive permissions create \
  --params '{"fileId": "FILE_ID"}' \
  --json '{"type": "user", "role": "reader", "emailAddress": "teammate@example.com"}'

Part 6

GitHub + Skill + gws를 하나의 요청으로 묶는다

좋은 통합 프롬프트는 작업, 제한, 검증, 외부 실행 승인 조건을 모두 포함한다.

핵심은 agent가 할 일과 사람이 승인할 일을 분리하는 것이다.

실습 6-A

통합 프롬프트 초안

첫 부분에는 저장소, issue, branch 규칙을 적는다. agent가 어디에서 어떤 기준으로 시작해야 하는지 알려주는 부분이다.

wel-bridge 저장소에서 issue #23 문서 작업을 진행해줘.
팀 Git Skill을 따라 branch와 PR 규칙을 지켜줘.

작업 전:
1. git status를 확인한다.
2. 현재 branch를 확인한다.
3. main에 직접 작업 중이면 docs/23 branch를 만든다.
실습 6-B

통합 프롬프트 실행 조건

두 번째 부분에는 검증과 제한을 적는다. 특히 외부 서비스 실행은 payload 확인 뒤 승인받도록 분리한다.

작업 후:
1. git diff를 요약한다.
2. PR 설명 초안을 작성한다.
3. 리뷰 일정을 만들 Calendar payload를 보여준다.
4. 변경 요약 파일을 Drive에 업로드할 명령을 보여준다.

제한:
- main에는 직접 push하지 마.
- commit, push, PR 생성은 내가 확인한 뒤 진행해.
- Calendar 생성과 Drive 업로드는 실행 전에 payload를 보여줘.
- credential, .env, token은 절대 commit하지 마.
최종 체크

실행 전 사람이 확인할 것

이 체크리스트는 실습 후에도 그대로 팀 작업 전에 사용한다.

  • 현재 branch가 main이 아닌가?
  • branch 이름에 issue 번호가 있는가?
  • git diff를 사람이 읽었는가?
  • PR 설명에 변경 사항, 검증, 리뷰 포인트, 관련 이슈가 있는가?
  • Calendar 참석자, 시간대, 설명을 확인했는가?
  • Drive 파일명, 폴더 ID, 공유 권한 role을 확인했는가?
  • worktree 또는 sparse checkout을 쓴다면 agent에게 작업 경로와 checkout 범위를 알려줬는가?
마무리

Agent에게 일을 맡길수록 규칙은 더 구체적이어야 한다

  • Git은 변경 이력을 통제한다.
  • PR은 팀 검토를 보장한다.
  • Skill은 팀 규칙을 반복 적용하게 만든다.
  • gws CLI는 Calendar와 Drive 작업을 명령과 payload로 관리하게 해준다.
